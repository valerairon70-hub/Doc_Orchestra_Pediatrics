# Doc Orchestra — Pediatrics
### AI-ассистент для врача педиатра-инфекциониста

---

## Что это за проект

Doc Orchestra — многоагентная AI-система, которая автоматизирует 80% рутинной коммуникации и документации педиатрического врача 24/7. Система получает сообщения от родителей, собирает жалобы через структурированный диалог, формирует SOAP-записи и подаёт готовые ответы врачу на одно нажатие кнопки. Цель — освободить 3–4 часа в день.

---

## Проблема — зачем это нужно

Врач педиатр-инфекционист в соло-практике ежедневно сталкивается с:

- **Сообщениями в 22:00–23:00** — родители пишут вечером, ожидают ответа срочно
- **Хаотичными описаниями** — «ребёнок плохо себя чувствует» вместо температуры, симптомов, длительности
- **Ручным сбором жалоб** — каждый раз одни и те же уточняющие вопросы
- **Одинаковыми рекомендациями** — паracetamol при температуре пишется 10 раз в день
- **Потерями в потоке** — важное сообщение тонет среди 50 других

Итог: 3–4 часа в день уходит на рутину, а не на медицину.

Doc Orchestra берёт эту рутину на себя. Врач получает структурированный пакет — и тратит 30 секунд на решение вместо 10 минут на переписку.

---

## Для кого

**Врач:** педиатр-инфекционист (соло-практика, ~20 сложных случаев в день).

**Родители пациентов:** пишут в мессенджер (WhatsApp/Telegram) — система отвечает как ассистент и готовит пакет для врача.

---

## Основные функции MVP (что работает сейчас)

- **Сбор жалоб** — AI ведёт диалог с родителем, задаёт уточняющие вопросы (симптомы, температура, длительность, возраст)
- **Экстренный режим** — «ребёнок задыхается» → мгновенный ответ «103!» без вызова API, даже при отключённом интернете
- **SOAP-заметка** — автоматически генерируется после сбора жалоб, врач видит структурированную карточку
- **Кокпит врача** — очередь запросов, просмотр диалога и SOAP, шаблоны быстрых ответов
- **Guardrail** — ни одно сообщение не уходит родителю без явного нажатия «Одобрить»
- **Telegram-уведомление врачу** — при новом запросе врач получает сообщение в Telegram (имя, симптом, ссылка)
- **Статус для родителя** — «✓ Сообщение получено» после отправки, «Ожидайте ответа врача» при передаче
- **Авторизация кокпита** — HTTP Basic Auth, пароль задаётся в `.env`
- **DEMO режим** — работает без API-ключа на заготовленных ответах

## Текущие страницы (работают сейчас — MVP)

| URL | Страница | Кто использует |
|-----|----------|----------------|
| `http://localhost:8081/parent` | Чат родителя | Родитель пишет жалобу |
| `http://localhost:8081/cockpit` | Кокпит врача | Врач одобряет/редактирует ответы |

### Как это работает прямо сейчас:
1. Родитель открывает `/parent` и описывает проблему
2. AI (Claude Haiku) ведёт сбор жалоб: симптомы → уточнения → передача врачу
3. Когда собрана вся информация, Claude добавляет внутренний маркер `[ГОТОВО]` — сервер генерирует SOAP и уведомляет кокпит
4. В кокпите `/cockpit` врач видит диалог и SOAP-заметку
5. Врач нажимает **Одобрить** — ответ отправляется родителю
6. Без одобрения врача — ни одно сообщение не уходит (жёсткий guardrail)

**API эндпоинты:**
- `GET /api/sessions` — список активных сессий (страховка: кокпит подгружает при открытии)

## Архитектура агентов (полная версия)

```
Родитель
   ↓
[Dialogue Agent]  — сбор жалоб (FSM: 5 фаз)
   ↓
[Orchestrator]    — маршрутизатор и guardrail
   ├── [Vision Agent]       — фото кожи + PDF анализов
   ├── [Mx Agent]           — план лечения по гайдлайнам (NICE/BMJ)
   └── [Longitudinal Agent] — тренды гемоглобин/рост + Red Flags
         ↓
[Cockpit]  — врач одобряет → ответ родителю
```

## Как запустить MVP

```bash
# 1. Активировать виртуальное окружение
source .venv/bin/activate

# 2. Установить зависимости (если первый раз)
pip install -r requirements.txt

# 3. Создать .env файл
cp .env.example .env
# Вставить ANTHROPIC_API_KEY и при необходимости сменить COCKPIT_PASSWORD

# 4. Запустить MVP
python mvp.py

# 5. Открыть в браузере:
#    Родитель: http://localhost:8081/parent
#    Врач:     http://localhost:8081/cockpit  (логин: любой, пароль: из COCKPIT_PASSWORD)
```

## Как проверить что система работает (ручной тест)

1. Запустить `python mvp.py`
2. Открыть `/cockpit` в одной вкладке → ввести пароль
3. Открыть `/parent` в другой вкладке
4. Написать: `"У ребёнка температура 38.5"`
5. Ответить на вопросы ассистента (длительность, возраст)
6. После финального сообщения ассистента — в кокпите должен появиться запрос
7. В кокпите нажать **Одобрить** → родитель должен получить ответ врача

**Проверка экстренного режима:**
- Написать в чате: `"ребёнок задыхается"` → мгновенный ответ "103!" без задержки

**Проверка без API (DEMO режим):**
- Убрать `ANTHROPIC_API_KEY` из `.env` → система работает на заготовленных ответах

## Обработка ошибок

| Ситуация | Что происходит | Что видит пользователь |
|----------|---------------|----------------------|
| **Claude API недоступен** | `get_ai_response()` перехватывает исключение | Родитель: *"Извините, произошла техническая ошибка. Обратитесь к врачу напрямую."* |
| **SOAP не сгенерировался** | `generate_soap()` возвращает текст ошибки | Врач: в панели SOAP — *"Ошибка генерации SOAP: ..."*, диалог всё равно доступен во вкладке «Диалог» |
| **Потеря WebSocket (родитель)** | Клиент автоматически переподключается через 2.5 сек | Строка статуса: *"Переподключение..."*, затем чат восстанавливается |
| **Потеря WebSocket (врач)** | Кокпит отключается от сервера | При переподключении сервер отправляет все текущие кейсы через `init`, кокпит подгружает их через `/api/sessions` |
| **Перезапуск сервера** | Все сессии в памяти теряются | Родитель при переоткрытии видит старые сообщения из localStorage, но ассистент начнёт заново; врач видит пустую очередь |
| **Неверный пароль кокпита** | FastAPI возвращает HTTP 401 | Браузер показывает диалог входа снова |
| **Экстренный случай** | Ответ формируется локально, **без вызова API** | Мгновенное *"Вызовите 103!"* — работает даже если Claude недоступен |
| **Родитель пишет пока ждёт врача** | Сервер блокирует новые сообщения | *"Ваш запрос уже передан доктору. Ожидайте ответа."* |

### Известные ограничения

- **Сессии не персистентны** — перезапуск сервера стирает все активные разговоры. Решение: PostgreSQL (Фаза 3).
- **Нет профиля пациента** — каждый разговор начинается с нуля, история не сохраняется. Решение: база данных (Фаза 3).
- **SOAP при ошибке API** — врач всё равно видит диалог и может написать ответ вручную.

## Дизайн-стиль

- **Цветовая схема:** тёмная тема (медицинский профессиональный вид)
- **Кокпит врача:** минималистичный, информация без лишнего — только то, что нужно для решения
- **Чат родителя:** светлый, дружелюбный, WhatsApp-стиль (белые пузыри от ассистента, зелёные от родителя)
- **Типографика:** системные шрифты, без украшений — читаемость на первом месте
- **Приоритет:** функция > красота. Врач должен работать быстро, родитель — не пугаться

## Чего НЕ будет (пока) — сознательные ограничения

Это важно для фокуса. Мы намеренно не делаем:

| Что | Почему нет |
|-----|------------|
| ❌ Автоответы родителям без врача | Безопасность — врач всегда в контуре |
| ❌ Постановка диагнозов | Юридически и этически недопустимо для AI |
| ❌ Педиатры других специальностей | Сначала один врач, потом масштаб |
| ❌ Интеграция с МИС / ЭМК | Сложно, не нужно для MVP |
| ❌ Мобильное приложение | Веб достаточно для проверки гипотезы |
| ❌ Назначение лекарств | Только информация о дозах — решение за врачом |
| ❌ Работа без интернета | Облачные AI модели требуют подключения |
| ❌ Пациенты старше 18 лет | Педиатрическая специализация — своя логика доз и норм |

---

## Ключевой принцип безопасности

> **Ни одно сообщение родителю не уходит без явного одобрения врача.**
> Это не просто инструкция в промпте — это жёсткий код в WebSocket-обработчике кокпита (`mvp.py`): сообщение отправляется только когда врач явно нажимает «Одобрить».

## Следующие шаги

Актуальный порядок работы (см. подробности в `PLAN.md`):

1. **Настроить Telegram-уведомления** — добавить `TELEGRAM_BOT_TOKEN` и `DOCTOR_CHAT_ID` в `.env` (инструкция в `.env.example`)
2. **Показать MVP врачу** — собрать обратную связь, уточнить топ-5 жалоб и шаблоны ответов
3. **Автоответ в нерабочие часы** — «Врач ответит утром до 9:00» при обращении ночью
4. **PostgreSQL + профиль пациента** — история обращений, до первого реального использования
5. **Telegram для родителей** — родители пишут из мессенджера, не из веб-чата
